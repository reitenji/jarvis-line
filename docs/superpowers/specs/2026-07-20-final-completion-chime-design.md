# Final Completion Chime

## Goal

Make final speech immediately distinguishable from commentary by playing one
short, gentle notification chime immediately before the final Jarvis Line.
Keep the behavior lightweight, cross-platform, and optional.

## Considered Approaches

### Operating-System Notification Sound

Use a named macOS, Windows, or Linux system sound. This is small to implement,
but sound names, loudness, availability, and character vary by platform and
installation. It would not give Jarvis Line a consistent completion cue.

### Bundled Audio Asset

Ship a WAV file with the Python package and macOS app. Playback would be
consistent, but the binary asset adds packaging and release-surface work and
can be misplaced by source-only installations.

### Procedural WAV

Generate a tiny WAV in memory with Python's standard library, play it through
the existing cross-platform player, and delete its temporary file immediately.
This is the selected approach because it has no dependency or persistent asset,
works for every TTS backend, and keeps the sound consistent across platforms.

## Behavior

- The chime is enabled by default for existing and new installations.
- It plays only for normalized `final` jobs, never for commentary, attention,
  warm-up, or ordinary TTS tests.
- It plays immediately before final speech while the existing global audio lock
  is held, so it cannot overlap another Jarvis Line playback.
- Stale, expired, cancelled, deduplicated, or replaced jobs never produce a
  chime because they do not reach playback.
- A chime playback error is logged without spoken content and does not prevent
  primary or fallback TTS from speaking.
- One final job produces at most one chime, including when primary TTS falls
  back to another backend.

## Sound Design

The cue is a low-amplitude, approximately 300 ms ascending two-tone chime with
short attack and release envelopes. Amplitude is baked into the generated WAV
so Windows, macOS, and Linux hear the same restrained level even when their
players expose different volume controls. The waveform is cached in memory for
the worker lifetime; each playback uses a securely created temporary WAV that
is removed in a `finally` block.

## Configuration And App

Add `final_chime_enabled`, a boolean with default `true`, to the shared config
contract and every backend capability. Existing config files inherit the new
default through the normal defaults merge.

The macOS app adds a **Final chime** switch under Speech > Events. The control
is disabled when speech is disabled or speak mode is off. No free-form sound,
path, or volume fields are exposed.

## Architecture

A small Python module owns deterministic WAV generation. The audio worker owns
policy and playback: `speak_line` receives the job phase, checks the setting,
plays the chime inside the audio lock, then invokes the selected TTS backend.
The queue schema and event protocol do not change.

The Swift config draft reads, validates, saves, and displays the boolean using
the existing settings flow. No additional process, thread, timer, model load,
or long-lived file is introduced.

## Verification

- Unit-test that generated bytes are a valid, bounded mono PCM WAV.
- Unit-test that a final phase plays the chime before speech.
- Unit-test that commentary and attention do not play it.
- Unit-test that disabling the setting suppresses it.
- Unit-test that a chime exception still allows TTS playback.
- Cover the new default and macOS config round trip.
- Run the complete Python suite, smoke test, quick soak, and macOS Swift tests.
- Play the chime locally once before installing the updated app/runtime.
